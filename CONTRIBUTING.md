# Contributing to Bluesound Dashboard

Thanks for contributing. This guide covers bugs, features, local setup, and pull requests.

Please read and follow the [Code of Conduct](CODE_OF_CONDUCT.md).

## How to contribute

### Reporting bugs

1. Check [Issues](https://github.com/tbaur/bluesound-dashboard/issues) for an existing report.
2. Open a new issue with the **Bug Report** template. Include:
   - Dashboard version (footer or `backend/app/__init__.py`)
   - OS, Python, Node, and browser (if UI)
   - Steps to reproduce, expected vs actual
   - Relevant logs (redact anything sensitive)

### Suggesting features

1. Check existing issues first.
2. Open a **Feature Request** with the problem, proposed solution, and use cases.

### Pull requests

1. Fork (if needed) and create a branch: `git checkout -b feature/short-description`
2. Make focused changes with tests for new behavior.
3. Run the checks below.
4. Open a PR with a **Conventional Commit** title (see below).
5. Keep PRs small and reviewable.

#### Commit / PR titles

PR titles drive automated releases via [release-please](RELEASING.md):

| Prefix | Meaning |
|--------|---------|
| `feat:` | New feature (minor) |
| `fix:` | Bug fix (patch) |
| `docs:`, `test:`, `refactor:`, `chore:`, `ci:` | No release |

Example: `feat: add multi-room group create`

`CHANGELOG.md` is generated by release-please from Conventional Commit / PR titles. Do not edit it by hand for routine releases.

#### PR checklist

- [ ] Tests added/updated when behavior changes
- [ ] Backend and/or frontend checks pass locally
- [ ] Documentation updated if needed
- [ ] Descriptive Conventional Commit PR title

## Development setup

### Prerequisites

- Python 3.10+ (CI: 3.10, 3.13, 3.14)
- Node.js 22+
- Git

### Setup

```bash
git clone git@github.com:tbaur/bluesound-dashboard.git
cd bluesound-dashboard
make install
```

That creates `backend/.venv`, installs the backend with dev extras, and runs `npm ci` in `frontend/`. Copy [.env.example](.env.example) to `.env` in the repo root when you need non-default settings. See [docs/CONFIGURATION.md](docs/CONFIGURATION.md) and [docs/RUNBOOK.md](docs/RUNBOOK.md).

### Run locally

```bash
make run
```

Opens the UI at http://127.0.0.1:8765/ after the API is healthy on `:8000`. For a production UI bundle: `make build` (writes `frontend/dist`). See [docs/RUNBOOK.md](docs/RUNBOOK.md).

### Running checks

```bash
# Backend
cd backend
ruff check app tests
mypy app
pytest  # requires ≥90% coverage (`--cov-fail-under=90`)

# Frontend
cd frontend
npm run lint
npm run typecheck
npm test
npm run build
```

## Project structure

```
bluesound-dashboard/
├── Makefile              # make install / build / run
├── scripts/              # install, build, run implementations
├── backend/app/          # FastAPI app (discovery, BluOS client, API, poller)
├── backend/tests/        # Backend tests (≥90% coverage required in CI)
├── frontend/src/         # React UI (Vite on :8765)
├── frontend/tests/       # Frontend unit tests
├── docs/                 # Configuration + runbook
├── docs/images/          # README screenshots
├── CODE_OF_CONDUCT.md
├── SECURITY.md
└── .github/              # CI, Dependabot, issue/PR templates
```

## Code standards

- Validate all external input at API boundaries.
- Keep BluOS protocol knowledge in `backend/app/bluos/`.
- Do not hardcode device IPs — discovery owns the fleet.
- Do not log secrets or full BluOS payloads.
- Prefer small functions, typed interfaces, and tests for new paths.
